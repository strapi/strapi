# i18n

End-to-end coverage for Strapi's internationalisation (i18n) plugin, exercising locale management, per-locale content authoring, permissions, and bulk publishing flows from the admin panel.

## Overview

The i18n specs drive Playwright against the admin panel to verify that content editors can organise multilingual content reliably. They cover settings-level locale CRUD, list-view and edit-view integrations (locale column, locale switcher, locale-aware URL parameters), single- and collection-type localisation, draft and publish semantics per locale, isolation of localised vs non-localised fields, role-based permissions scoped to individual locales, bulk publish/unpublish modals across locales, and the "Fill in from another locale" helper including how it handles localised and non-localised relations. Every spec resets the database from the `with-admin` data transfer fixture before each test to ensure deterministic state.

## Test specs

### `i18n/bulk-locale-modal.spec.ts` — Bulk locale modal

**Purpose:** Verify that the bulk locale actions modal enumerates every locale on a document, including newly created localisations.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.
- Test suite timeout raised to 500000 ms.

**Scenarios covered:**

- `Bulk locale actions` -> `Bulk locale modal lists all locales after creating a new localization`: Opens the Article "Why I prefer football over soccer", publishes the English entry, creates a Spanish (es) localisation, then opens the "Unpublish multiple locales" modal and confirms both locales appear and can be selected and unpublished successfully.

### `i18n/create-edit.spec.ts` — Create and edit operations

**Purpose:** Exercise the two primary authoring flows for localised documents: creating a brand new non-default-locale document, and adding a new locale entry to an existing document.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.
- Suite timeout raised to 500000 ms.
- `afterAll` restores modified files with `resetFiles()`.

**Scenarios covered:**

- `Create and Edit Operations` -> `As a user I want to create a brand new document in the non-default locale`: Switches the Products list-view to Spanish (es), creates a new entry, asserts the locale query string is propagated to UID generation, publishes the document, then verifies switching the edit-view back to English shows "Untitled" (a separate document).
- `Create and Edit Operations` -> `As a user I want to add a locale entry to an existing document`: Opens an existing English product, switches the edit-view locale combobox to Spanish (es), authors and publishes the localised version, and confirms that both locales belong to the same document (switching back to English still shows the original title).

### `i18n/editview.spec.ts` — Edit view

**Purpose:** Cover edit-view level i18n behaviour including localised authoring, permission-gated creation, per-locale deletion for single and collection types, bulk publish/unpublish modals invoked from the edit view, pre-filling of non-translatable fields, and the optional unstable AI translation feature toggle.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.
- Suite timeout raised to 500000 ms.
- The AI-translation describe blocks only run when `STRAPI_FEATURES_UNSTABLE_AI_LOCALIZATIONS=true`.

**Scenarios covered:**

- `Edit view` -> `As a user I want to create a brand new document in the non-default locale`: Mirrors the create-edit flow at the edit-view level - switches to Spanish, creates a Product, verifies UID generation per locale and that swapping back to English reveals a separate "Untitled" document.
- `Edit view` -> `As a user I want to add a locale entry to an existing document`: Adds a Spanish localisation to the existing Nike product, publishes it, and confirms the two locales share one underlying document.
- `Edit view` -> `As a user I should not be able to create a document in a locale I don't have permissions for`: Edits the Editor role to grant full English (en) permissions but revoke French (fr) Create, logs back in as the editor, succeeds at creating an English article, and asserts "Create French (fr) locale" is disabled.
- `Edit view` -> `As a user I should be able to delete a locale of a single type and collection type`: Creates English and Spanish entries for both an Article (collection type) and the Homepage (single type), then uses "Delete entry (Spanish (es))" from the more-actions menu to remove just the Spanish locale of each.
- `Edit view` -> `As a user I want to publish multiple locales of my document`: Creates a Spanish draft alongside an existing English article, opens "Publish multiple locales", selects all, confirms "2 entries ready to publish", publishes, and asserts both rows become "Already Published" with the publish button disabled.
- `Edit view` -> `As a user I want to unpublish multiple locales of my document`: Publishes both English and Spanish variants of an article, opens "Unpublish multiple locales", selects all, confirms "Ready to unpublish" count of 2, unpublishes, and asserts both rows return to "Draft" with the button disabled.
- `Edit view` -> `As a user I want to see non translatable fields pre-filled when creating a new locale of a document`: Uses the Content-Type Builder to add a non-translatable text field to Products, creates an English entry, switches to French, and verifies the non-translatable field pre-fills while the translatable `name` field stays empty. Cleans up by removing the field.
- `Edit view (AI, unstable)` -> `As a user I want to enable AI translation`: When the unstable AI-localisations feature flag is set, opens an Article, confirms the "AI Translation Status" tooltip reads "disabled", toggles the setting in `/admin/settings/internationalization`, and confirms the tooltip now reads "enabled".
- `Unstable edit view` -> `As a user I want to enable AI translation`: Duplicate of the above scenario under an additional `Unstable edit view` describe wrapper, also gated by the same feature flag.

### `i18n/listview.spec.ts` — List view

**Purpose:** Confirm that list-view integrations (locale column, locale switcher) only appear for i18n-enabled content types, and that the selected locale persists across navigation.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.

**Scenarios covered:**

- `List view` -> `As a user I want a locale column to be injected in the table & a locale switcher only if the content-type has i18n enabled.`: Navigates to the i18n-enabled Products list-view, asserts the "Available in" column and locale switcher are present with all four seeded locales (English, French, German, Spanish), then navigates to the non-i18n Author type and asserts neither element renders.
- `List view` -> `As a user I want to be able to navigate from one localized content-type to another and persist the last selected locale`: Switches Products to Spanish (es) and creates an entry, then verifies the `plugins[i18n][locale]=es` query parameter persists when navigating to Articles (localised), to Authors (non-localised but still carrying the param), and back to Products.

### `i18n/localized-fields.spec.ts` — Localised fields (locale isolation)

**Purpose:** Prove that localised fields are isolated per locale while non-localised fields propagate across all locales, and that publishing state is per-locale.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.
- Suite timeout raised to 500000 ms.
- `afterAll` restores files with `resetFiles()`.

**Scenarios covered:**

- `Locale Isolation` -> `As a user I want to verify that modifying a localized field only affects the current locale`: Creates a Spanish localisation of the Nike product, modifies the English `name` field, and confirms the Spanish `name` is unchanged while the English edit persists.
- `Locale Isolation` -> `As a user I want to verify that publishing only affects the current locale`: Creates a Spanish draft, publishes only the English locale, then confirms via the list-view that the English row shows status "published" while the Spanish row shows "draft".
- `Locale Isolation` -> `As a user I want to verify that modifying a non-localized field affects all locales`: Unchecks the non-localised `isAvailable` field in English, saves, then switches to Spanish and confirms the checkbox is also unchecked - demonstrating propagation across locales.

### `i18n/permissions.spec.ts` — Locale permissions

**Purpose:** Validate that role-based permissions can be scoped per locale and that locale-scoped delete actions work for both collection types and single types.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.
- Suite timeout raised to 500000 ms.
- `afterAll` restores files with `resetFiles()`.

**Scenarios covered:**

- `Locale Permissions` -> `As a user I should not be able to create a document in a locale I don't have permissions for`: Grants the Editor role full English permissions on Article but revokes French Create, logs in as the editor, creates an English article successfully, then asserts "Create French (fr) locale" is disabled.
- `Locale Permissions` -> `As a user I should be able to delete a locale of a single type and collection type`: Creates English and Spanish entries for an Article (collection type) and Homepage (single type), then deletes only the Spanish locale of each via the "Delete entry (Spanish (es))" menu action.

### `i18n/settings.spec.ts` — Settings

**Purpose:** Cover locale CRUD from the Internationalisation settings page and verify that changes propagate through list-views, edit-views, and related UI such as dynamic-zone validation.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture, permissions pruned via `prunePermissions`, and admin login before each test.
- Behaviour branches on `STRAPI_DISABLE_EE` to assert either EE or CE document-action menu states.

**Scenarios covered:**

- `Settings` -> `As a user I want to create a locale and then create an entry in that locale`: Opens Settings -> Internationalisation, adds Italian (it), then navigates to the Shop single type, switches to the new locale, authors a title, adds required dynamic-zone components (Product carousel, Hero image) to satisfy a `min: 2` constraint, and publishes. Asserts CE vs EE differences in the "More document actions" menu state for a brand-new localisation.
- `Settings` -> `As a user I want to delete an existing locale and have the content deleted as well`: Confirms the Articles list-view initially exposes all four locales, deletes French from settings, then confirms French has disappeared from the locale dropdown while other locales and the English articles remain.
- `Settings` -> `As a user I want to update a locale and have the changes reflected across my application`: Renames the default English (en) locale's display name to "UK English" from settings, then verifies the Products list-view reflects the new display name in the locale combobox and options.

### `i18n/fill-from-locale.spec.ts` — Fill from another locale

**Purpose:** Validate the "Fill in from another locale" helper, especially how it handles localised relations that may or may not exist in the target locale, and non-localised relations.

**Preconditions:**

- Database reset from the `with-admin` DTS fixture and admin login before each test.

**Scenarios covered:**

- `i18n - Fill from another locale` -> `As a user I want to retrieve localized relations with the fill from another locale feature`: Three consecutive cases inside one test. First, on the Shop single type with a product relation that does not yet exist in Spanish, "Fill in from another locale" copies the English title but the relation is absent. Second, after creating a Spanish localisation of that product, running "Fill in from another locale" on Shop now copies both the title and the relation. Third, on the Article type it confirms that "Fill in from another locale" also pulls in non-localised relations such as the Author "Coach Beard".
